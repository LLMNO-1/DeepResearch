# 环境配置问题排查与修复

## 问题一：MongoDB 认证失败

### 报错

```
pymongo.errors.OperationFailure: Command insert requires authentication
pymongo.errors.OperationFailure: Authentication failed.
```

### 原因

虚拟机 `192.168.10.140` 上的 MongoDB 容器启动了认证，但 `.env` 里的连接串没有带用户名密码：

```
MONGODB_URI=mongodb://192.168.10.140:27017
```

容器环境变量：

| 变量 | 值 |
|---|---|
| MONGO_INITDB_ROOT_USERNAME | admin |
| MONGO_INITDB_ROOT_PASSWORD | 123456 |

应用初始化脚本 `mongo/init-mongo.js` 创建了应用用户 `deepresearch` / `deepresearch_dev`，认证库为 `deep_research`。但容器数据卷已有历史数据，init 脚本不会重新执行，该用户未创建，手动执行 mongosh 创建了用户。

### 修复

将 URI 改为带认证的格式：

```
MONGODB_URI=mongodb://deepresearch:deepresearch_dev@192.168.10.140:27017/deep_research?authSource=deep_research
```

## 问题二：DeepSeek API Key 未生效

### 报错

```
ValidationError: 1 validation error for ChatDeepSeek
Value error, If using default api base, DEEPSEEK_API_KEY must be set.
```

### 原因

pydantic-settings 读取 `.env` 文件后，只把值赋给了 Settings 类的 Python 属性（如 `settings.deepseek_api_key`），**不会自动注入到系统环境变量** `os.environ`。

而 LangChain 的 `ChatDeepSeek` 底层是通过 `os.environ['DEEPSEEK_API_KEY']` 直接读环境变量的，它感知不到 pydantic-settings 的存在。于是就出现：`.env` 里明明写了 key，Python 代码也读到了，但 LangChain 报错说没设置。

```
┌─────────────┐     ┌──────────────────┐     ┌─────────────────────┐
│   .env 文件   │ ──▶ │ pydantic-settings │ ──▶ │ Settings.deepseek   │
│ DEEPSEEK_    │     │ 只赋值给模型字段    │     │ _api_key = "sk-xxx" │
│ API_KEY=sk.. │     └──────────────────┘     └─────────────────────┘
└─────────────┘                                        ✗ 没有连通
                                                    ┌─────────────────────┐
                                              ──▶ │ os.environ          │
                                              ✗    │ 读不到 DEEPSEEK_     │
                                                   │ API_KEY              │
                                                   └─────────────────────┘
                                                            │
                                                    ┌───────▼─────────────┐
                                                    │ LangChain           │
                                                    │ ChatDeepSeek        │
                                                    │ 报错：KEY must be set│
                                                    └─────────────────────┘
```

另外，`.env` 文件中 `DEEPSEEK_API_BASE` 和 `DEEPSEEK_API_KEY` 前面有多余的空格缩进，导致 pydantic-settings 把空格当作变量名的一部分，解析出的值也是空的。这个问题修复后错误依然存在，因为根因是 pydantic-settings 没有把值注入 `os.environ`。

### 修复

在 `app/config/config.py` 的 `get_settings()` 中添加 `_export_to_env()`，将 LLM 密钥等关键配置同步注入 `os.environ`：

```python
def _export_to_env(settings: Settings) -> None:
    for field_name, env_var in [
        ("openai_api_base", "OPENAI_API_BASE"),
        ("openai_api_key", "OPENAI_API_KEY"),
        ("deepseek_api_base", "DEEPSEEK_API_BASE"),
        ("deepseek_api_key", "DEEPSEEK_API_KEY"),
        ("tavily_api_key", "TAVILY_API_KEY"),
    ]:
        value = getattr(settings, field_name, None)
        if value is not None:
            os.environ.setdefault(env_var, str(value))
```

修复后的数据流：

```
┌─────────────┐     ┌──────────────────┐     ┌─────────────────────┐
│   .env 文件   │ ──▶ │ pydantic-settings │ ──▶ │ Settings.deepseek   │
│ DEEPSEEK_    │     │ 赋值给模型字段      │     │ _api_key = "sk-xxx" │
│ API_KEY=sk.. │     └──────────────────┘     └──────────┬──────────┘
└─────────────┘                                          │
                                                  _export_to_env()
                                                          │
                                                          ▼
                                                   ┌─────────────────────┐
                                                   │ os.environ          │
                                                   │ DEEPSEEK_API_KEY    │
                                                   │ = "sk-xxx"          │
                                                   └──────────┬──────────┘
                                                              │
                                                              ▼
                                                   ┌─────────────────────┐
                                                   │ LangChain           │
                                                   │ ChatDeepSeek        │
                                                   │ 读取成功 ✓           │
                                                   └─────────────────────┘
```

## 问题三：Windows 下 UnicodeEncodeError（ASCII 编码器无法处理中文）

### 报错

```
UnicodeEncodeError: 'ascii' codec can't encode characters in position 10-13:
ordinal not in range(128)
During task with name 'tools' and id '2c574b1c-357d-f56a-53ad-a46d3eae7289'
During task with name 'tools' and id 'd5a19489-e8a7-0d99-9a01-aede51f13eed'
```

### 原因

Windows 终端默认编码是 GBK（`sys.stdout.encoding = 'gbk'`），不是 UTF-8。LangChain/LangGraph 底层使用 asyncio 创建工具调用任务时，任务载荷和工具返回的中文字符经过内部 `str()` 或序列化操作，走了 ASCII 编码器：

```
Python 启动
    │
    ├── sys.stdout.encoding = 'gbk'
    ├── sys.stderr.encoding = 'gbk'
    │
    ▼
LangGraph 创建 asyncio 任务
    │
    ├── 任务名: 'tools'
    ├── 载荷包含中文: "研究边界和核心问题..."
    │
    ▼
内部序列化 / print / repr
    │
    └── 走 ASCII 编码器 → 中文无法编码 → UnicodeEncodeError
```

Python 默认编码虽然是 `utf-8`，但 I/O 流（stdout/stderr）走终端编码 GBK，部分第三方库在 Windows 上内部处理字符串时会回退到 ASCII。

### 修复

**改动一**：`app/main.py` — 在任何模块导入前，把 stdout/stderr 重新包装为 UTF-8：

```python
import sys
import io

sys.stdout = io.TextIOWrapper(sys.stdout.buffer, encoding="utf-8", errors="replace")
sys.stderr = io.TextIOWrapper(sys.stderr.buffer, encoding="utf-8", errors="replace")
```

**改动二**：`app/config/config.py` 的 `_export_to_env()` — 注入 UTF-8 环境标识：

```python
os.environ.setdefault("PYTHONIOENCODING", "utf-8")
os.environ.setdefault("PYTHONUTF8", "1")
```

**改动三**：启动命令必须前置 `PYTHONIOENCODING`（该变量必须在 Python 进程启动前设定，否则不生效）：

PowerShell：
```
$env:PYTHONIOENCODING='utf-8'; uv run uvicorn app.main:app --reload --host 0.0.0.0 --port 8000
```

CMD：
```
set PYTHONIOENCODING=utf-8 && uv run uvicorn app.main:app --reload --host 0.0.0.0 --port 8000
```

### 为什么改动一无法独立生效

`io.TextIOWrapper` 重包装只影响 `sys.stdout`/`sys.stderr` 的显式输出。LangChain/LangGraph 内部 asyncio 任务的错误消息并非全部经过 `sys.stdout` 打印——部分异常信息由 asyncio event loop 的 `call_exception_handler` 直接格式化输出到控制台，走的是底层 Win32 API，不受 Python 层 stdout 包装控制。因此 `PYTHONIOENCODING` 是兜底方案。
