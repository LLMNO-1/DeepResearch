## 一、项目定位：chatbot vs 深度研究平台

课件用一张对比表阐明了这个项目和普通聊天机器人的本质区别：

| 对比项 | 普通 chatbot | 深度研究报告平台 |
|--------|-------------|-----------------|
| 核心对象 | 对话消息 | 研究项目 |
| 用户输入 | 即时问题 | 研究主题和研究设定 |
| 系统过程 | 直接生成回答 | 生成大纲 → 确认大纲 → 执行研究 → 生成报告 |
| 状态管理 | 对话历史 | 项目状态、任务状态、报告版本 |
| 信息来源 | 模型知识或临时检索结果 | 外部公开资料、内部知识库 |
| 输出结果 | 一段回答 | HTML 报告、事实卡片、洞察卡片、引用来源 |
| 可靠性要求 | 用户自行判断 | 结论需要能追溯到来源 |

**核心差异**：chatbot 是"一问一答"的无状态对话；深度研究平台是"多阶段推进"的有状态项目——项目状态、大纲版本、任务进度、报告版本都需要持久化管理。

## 二、研究过程的 6 个环节拆解

课件把传统研究流程拆成了明确的系统链路：

```
研究问题
  → 研究项目（创建项目、保存初始状态）
  → 研究任务书（明确目标、范围、假设、成功标准）
  → 研究大纲（多级章节树，每个节点有 title/question/description）
  → 资料检索（公开搜索 + 网页读取 + 内部知识库）
  → 事实卡片（可复核陈述 + 来源 + 置信度）
  → 洞察卡片（基于事实的判断 + 支撑事实 ID）
  → 结构化研究结果（所有章节 + 来源 + 事实 + 洞察）
  → HTML 报告（确定性渲染，不调用 LLM）
```

每个环节的产出都是下一个环节的输入，形成清晰的流水线。

## 三、状态机设计

课件用电商订单类比解释了状态机概念：
- **状态**：对象在某一时刻所处的阶段
- **事件**：触发状态变化的外部动作
- **转移**：在某个状态下、发生某个事件后、去往下一个状态

### 项目状态机

```
[*] → created
created → brief_generating: 创建研究项目
brief_generating → outline_ready: 大纲生成成功
brief_generating → brief_generating: 大纲生成失败后可重试
outline_ready → outline_revising: 用户提交修改意见
outline_revising → outline_ready: 大纲修改成功
outline_ready → outline_confirmed: 用户确认大纲
outline_confirmed → research_running: 提交报告生成任务
research_running → report_ready: 研究和报告生成成功
report_ready → completed: 用户归档或完成
```

### 任务状态机

```
[*] → queued → running → succeeded
                       → failed
```

### 为什么项目状态和任务状态要分开

一个项目可以有多个任务。例如：

| 任务类型 | 触发动作 | 项目状态变化 |
|---------|---------|------------|
| `generate_research_brief` | 创建研究项目 | `brief_generating` → `outline_ready` |
| `revise_outline` | 用户要求修改大纲 | `outline_revising` → `outline_ready` |
| `generate_report` | 用户提交报告生成任务 | `research_running` → `report_ready` |
| `render_report` | 基于已有研究结果重新渲染 | `report_ready` → `report_ready` |

前端根据项目状态决定展示什么 UI、什么按钮可以点击；根据任务状态决定 loading 动画和轮询节奏。

## 四、asyncio.create_task 详解

课件用代码示例对比了 `await` 和 `asyncio.create_task` 的区别：

```python
# await 方式：顺序执行，A 完成才执行 B
result_a = await fetch_data("A", 2)  # 等 2 秒
result_b = await fetch_data("B", 1)  # 再等 1 秒，总耗时 3 秒

# create_task 方式：并发执行
task_a = asyncio.create_task(fetch_data("A", 2))
task_b = asyncio.create_task(fetch_data("B", 1))
# 两者同时运行，总耗时 2 秒
```

在本项目中，路由层不 `await` 长耗时任务，而是 `asyncio.create_task` 提交后立即返回 `task_id`。

### 与 Celery 的对比

| 方案 | 优点 | 局限 | 适用阶段 |
|------|------|------|---------|
| `asyncio.create_task` | 简单、无额外组件 | 进程重启任务丢失、不适合多实例调度 | 第一版 MVP |
| Celery | 任务可排队、可重试、可分布式 | 需要 Redis/RabbitMQ 中间件 | 生产化改造 |

## 五、MongoDB 基础概念

课件对 MongoDB 做了入门级介绍，适合不熟悉文档数据库的读者：

| 概念 | 含义 | 类比关系型数据库 |
|------|------|-----------------|
| database | 数据库 | database |
| collection | 集合 | table |
| document | 文档 | row |
| field | 字段 | column |
| `_id` | 文档主键 | primary key |

### 为什么选文档数据库

研究项目天然是嵌套结构——一个大纲下面有多个章节，每个章节又有关键发现、证据链、表格、图表。用关系型数据库需要拆成多个表 + JOIN，而文档数据库一个 document 就能装下整个项目所有数据。

### MongoDB 基本操作

```python
# 插入
await collection.insert_one(document)

# 查询一条
await collection.find_one({"project_id": project_id})

# 查询多条（带排序）
await collection.find_one({"project_id": project_id}, sort=[("version", -1)])

# 更新特定字段
await collection.update_one(
    {"task_id": task_id},
    {"$set": {"status": "succeeded", "updated_at": now}}
)

# 数组操作：去旧 + 加新
await collection.update_one(
    {"project_id": project_id},
    {"$pull": {"sections": {"section_id": section_id}}}
)
await collection.update_one(
    {"project_id": project_id},
    {"$push": {"sections": new_section}}
)
```

## 六、LLM 能力边界设计原则

课件反复强调一条核心原则：

> LLM 负责理解问题、拆解问题、整理事实和形成洞察；确定性代码负责接口、状态、数据保存和报告渲染。

具体到各阶段：

| 阶段 | 谁来做 | 原因 |
|------|--------|------|
| 生成大纲 | LLM（研究管理智能体） | 需要理解研究目标和范围 |
| 修改大纲 | LLM（研究管理智能体） | 需要理解自然语言修改意见 |
| 资料检索 | LLM 协调 + API 执行 | 搜索本身不需要 LLM，但判断搜什么、如何评估来源需要 LLM |
| 提取事实 | LLM（信息检索智能体） | 从非结构化文本中提取结构化事实 |
| 写章节正文 | LLM（研究管理智能体） | 需要语言组织和归纳能力 |
| 渲染 HTML | 确定性代码 | 结构转换不需要推理，LLM 可能新增事实破坏证据链 |

### 如果让 LLM 渲染报告会出什么问题

1. 报告阶段可能新增事实 → 来源和证据链断裂
2. 可能改写已有结论 → 前后不一致
3. HTML/CSS/目录/引用这些工作不需要 LLM 推理
4. LLM 自由生成 HTML 结构不可控，后端难以稳定落库

## 七、多智能体架构的难点

课件指出了多智能体系统的几个典型陷阱：

| 问题 | 结果 |
|------|------|
| 主智能体也搜索，检索智能体也写报告 | 职责混乱，输出不可控 |
| 报告阶段继续让 LLM 补内容 | 来源和事实链条断裂 |
| 所有结果一次性塞进上下文 | 内容过长，模型容易遗漏章节 |
| 只返回自然语言结果 | 后端无法稳定落库和渲染 |
| 工具返回异常没有结构化 | Agent 不知道应该重试、跳过还是降级 |

### 本项目的应对策略

- 主智能体只协调研究过程，不直接执行搜索
- 检索智能体只处理来源、事实和冲突，不写报告
- 主智能体必须通过 `save_research_section` 逐章节落库
- 报告渲染工具只做 HTML 展示转换
- 所有关键产物都使用 Pydantic 结构描述
- 工具异常返回结构化错误（`{"ok": false, "errors": [...]}`），Agent 可据此修正

## 八、可扩展的智能体方向

课件列举了后续可以扩展的 Agent（第一版不需要实现）：

- **问数智能体**：面向企业内部结构化数据库，负责 SQL 生成和指标查询
- **竞品分析智能体**：跟踪公司、产品、融资、价格、渠道信息
- **政策分析智能体**：专门检索政策文件、监管动态
- **财务分析智能体**：处理财报、经营数据和估值
- **图表规划智能体**：根据研究结果规划图表类型和结构

扩展原则：只有当某类任务有**独立工具、独立上下文和独立输出结构**时，才适合拆成单独 Agent。

## 九、端到端验证流程

课件提供了完整的 curl 验证步骤：

```bash
# 1. 健康检查
curl http://localhost:8000/health

# 2. 创建研究项目
curl -X POST http://localhost:8000/api/v1/research-projects \
  -H "Content-Type: application/json" \
  -d '{"topic":"研究具身智能行业未来三年的机会", ...}'
# 记录返回的 project_id 和 initial_task_id

# 3. 轮询大纲生成任务
curl http://localhost:8000/api/v1/tasks/{initial_task_id}
# 重复直到 status 变为 succeeded

# 4. 获取大纲
curl http://localhost:8000/api/v1/research-projects/{project_id}/outline

# 5. 确认大纲
curl -X PUT http://localhost:8000/api/v1/research-projects/{project_id}/outline \
  -H "Content-Type: application/json" \
  -d '{"action": "confirm"}'

# 6. 提交报告生成任务
curl -X POST http://localhost:8000/api/v1/research-projects/{project_id}/report-tasks \
  -H "Content-Type: application/json" \
  -d '{"user_instruction": "报告风格偏管理层汇报"}'
# 记录返回的 task_id

# 7. 轮询报告生成任务
curl http://localhost:8000/api/v1/tasks/{task_id}

# 8. 获取最新报告
curl http://localhost:8000/api/v1/research-projects/{project_id}/reports/latest
```

## 十、常见问题排查

| 问题 | 可能原因 | 排查方向 |
|------|---------|---------|
| 创建项目后一直没有大纲 | 后台任务未启动或 Agent 调用失败 | 查看任务状态接口和后台日志 |
| 报告任务返回 409 | 大纲未确认 | 先调用大纲确认接口 |
| 报告渲染失败 | research_result 缺失或结构不完整 | 检查项目文档中的 research_result 字段 |
| 报告来源为空 | 研究阶段未保存来源 | 检查章节 sources 和 evidence_chain |
| 报告返回 404 | 尚未成功保存报告版本 | 查询报告任务状态 |
| HTML 读不到 | 存储路径异常 | 检查 report_storage_local_dir 配置 |

## 十一、第一版系统边界

课件明确了当前版本**做**和**不做**的事情：

**已实现**：用户流程（创建→大纲→确认→报告→查看）、FastAPI 接口、Pydantic Schema、后台任务、MongoDB 持久化、双智能体协作、外部搜索/网页读取/RAGFlow/报告渲染工具、确定性 HTML 渲染

**暂不实现**：登录和权限、团队协作、复杂微服务、Celery 任务队列、独立报告写作 Agent、PDF 导出、文件上传、前端框架
