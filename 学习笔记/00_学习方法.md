# 学习方法

## 核心原则：按执行链路追代码，不按文件顺序读文档

不要从模块 1 顺着读到模块 11。而是选一个具体场景，跟着数据流走完整个调用链。

---

## 场景驱动：以一条请求为线索

以"用户点击创建研究项目"为例：

```
1. 前端发了什么请求
   → static/index.html（表单提交部分）

2. 请求到了后端的哪个路由
   → 04_API路由层（POST /research-projects 端点）
   → 02_Schemas（ResearchProjectCreate 的字段定义）

3. 路由调了 start_generate_research_brief_task，它做了什么
   → 05_后台任务调度（_schedule_task → asyncio.create_task）
   → 理解"为什么路由立即返回，任务在后台执行"

4. 后台任务调了 Agent
   → 06_Agent核心（ResearchAgent.generate_research_brief）
   → 07_Agent系统Prompt（Agent 收到了什么指令）

5. Agent 产出怎么存的
   → 03_数据访问层（save_research_brief_and_outline）
   → MongoDB 的 document 长什么样

6. 前端怎么知道任务做完了
   → 04_API路由层（GET /tasks/{task_id} 轮询）
   → 03_数据访问层（任务状态读写）

7. 回头看全局
   → 01_项目骨架（配置、main.py 是怎么把这一切组织起来的）
   → 11_课件补充（为什么设计成这样，而不是一个简单的 chatbot）
```

每一步都有具体的问题驱动——不是在"学 Repository 层"，而是在回答"Agent 的产出是怎么落库的"。

---

## 三遍阅读法

### 第一遍：追主链路（创建项目 → 大纲 → 确认 → 报告）

- 只看路由层 + 后台任务层的代码，理解数据怎么流动
- 搁置 Agent 内部细节、Prompt 细节、CSS 细节
- **目标**：能画出完整的时序图

### 第二遍：深入 Agent 层

主链路跑通后，钻进 Agent 核心、Prompt 工程、工具层这三篇。重点理解：
- Agent 门面模式怎么隔离框架细节
- Prompt 的职责边界是怎么通过"做什么/不做什么"来约束的
- 逐章节落库的循环逻辑（为什么最多 4 轮）

### 第三遍：补充细节

渲染引擎（报告怎么从结构化数据变成 HTML）、基础设施（Docker 部署）、课件补充中的设计原则和常见陷阱。

---

## 每篇文档怎么读

文档结构统一：先讲**职责定位**，再讲**关键代码**，最后讲**设计决策**。

读的时候抓三个问题：

1. **这层的输入是什么，输出是什么？**（层间衔接）
2. **这层做了什么决定？**（比如"工具函数不抛异常，返回错误字典"）
3. **如果不这样设计，会出什么问题？**（比如"如果让 LLM 渲染 HTML 会怎样"）

第三个问题最关键——理解"为什么"比理解"是什么"重要得多。

## 建议节奏

- 第一遍（主链路）：1-2 小时，边看文档边在 IDE 里 Ctrl+Click 跳转追代码
- 第二遍（Agent 层）：1 小时，重点读懂 Prompt 的 Few-shot 示例
- 第三遍（细节 + 课件补充）：1 小时

总共 3-4 小时能把整个项目的核心设计吃透。
